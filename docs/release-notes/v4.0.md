# LifeOS v4.0 Release Notes

## Release

**Version:** v4.0  
**Screenshot group:** Group 10 - LifeOS Spine Recovery Map  
**Build lane:** LifeOS spine completion  
**Status:** v4.0 code and Group 10 screenshots prove the corrected v4 road.

## What v4.0 changes

v4.0 corrects the roadmap after the v3.9 offline foundation.

v3.9 remains the offline/local-first foundation checkpoint.

v4.0 starts the spine completion road before v5 integrations.

The key correction is:

```text
v4 is not integrations yet.
v4 completes the LifeOS operating spine.
v5 starts real integrations.
v6.5 is the companion app checkpoint.
```

## Master rule

```text
Everything important becomes an item.
Every item has state.
Every state affects pressure.
Every pressure feeds the Command Centre.
```

## Why this matters

LifeOS is not just a dashboard and not just a set of modules.

LifeOS turns real-world pressure into reviewed, stateful items that link to:

- money
- time
- people
- work
- proof
- evidence
- timeline/context
- Universal Spine
- Command Centre

External data should not land directly into trusted state.

The correct flow is:

```text
External source / OCR / manual input
-> Integration Inbox / Sync Preview Queue
-> reviewed item
-> item state
-> evidence
-> timeline/context
-> pressure source
-> Command Centre signal
-> safe next action
```

## Boundary

v4.0 does not:

- connect real APIs
- request OAuth permissions
- sync email/calendar/accounting/bank data
- run AI actions
- build the companion app
- perform the major workspace redesign

Those come later.

## Current roadmap framing

```text
v3.9 = offline foundation checkpoint
v4.x = LifeOS spine completion
v5.x = integrations
v6.x = controlled automation / hardening
v6.5 = companion app
v7.x = AI assistant layer
v8-v9 = full mobile + major workspace reshape
```

## Result

v4.0 makes the original LifeOS spine explicit and visible in the app.

It gives v4.x a controlled road and gives v5 integrations somewhere safe to land.
