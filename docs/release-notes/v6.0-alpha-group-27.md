## Controlled automation foundation

Group 27 introduces visible automation intent without unattended execution.

Rules are disabled by default. Evaluation is manual and deterministic. Current inputs must be reviewed or trusted. Every condition displays the field checked, expected value, actual value, pass/fail result and trust state. A matched result becomes an explicit proposal with a target, risk level, approval requirement, execution mode and capability policy.

## Review boundary

The proposal queue supports explicit approval or rejection.

Approval records that the proposed intent was accepted as valid. It does not execute the operational action. Approved proposals remain visibly marked `executed: No`.

Rejected, approved, duplicate-suspected and blocked proposals remain retained with provenance and audit.

## Policy enforcement

Current safe internal proposal capabilities may be read and reviewed in dry-run mode.

The following capability classes remain blocked:

- external write
- communication send or mailbox mutation
- calendar mutation
- financial mutation
- destructive action
- arbitrary script or process execution
- plugin execution
- AI-generated rules or decisions

The fictional overdue-invoice email rule proves that a matching High-risk communication proposal is classified `BlockedByPolicy`, recorded in the queue and audit, and never executed.

## Duplicate handling

Repeated evaluation of the same rule against the same source-state snapshot uses a stable duplicate key. The repeated match links to the prior proposal instead of producing another clean actionable proposal.

## Persistence and audit

Rules, evaluations, proposals, decisions and audit records persist in local JSON storage using the existing LifeOS local-storage conventions and backup recovery behavior.
